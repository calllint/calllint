# 2. 第一性反思：现在最该补的不是功能，而是可信度

如果从第一性看，MCPGuard 的核心价值不是：

```text
我能发现很多风险
```

而是：

```text
开发者愿意相信我的裁决，并把它放进开发流程。
```

所以接下来要解决 4 个问题：

```text
1. 结果是否准确？
2. 结果是否可解释？
3. 结果是否可复现？
4. 结果是否能自然进入开发者工作流？
```

v0.2 已经解决了“能不能输出结果”。

下一步要解决的是：

```text
这个结果是否足够值得被 CI、IDE、团队规范采纳？
```

这是从工具脚本走向真正开发者产品的分水岭。

## 风险 5：online enrichment 容易引入“伪权威感”

`@mcpguard/online` 是可选、injectable fetch、测试 mock，这是正确设计。

但要非常小心：

```text
online metadata 只能 enrich
不能直接让 UNKNOWN → SAFE
不能让缺少 online 数据直接 BLOCK
不能让 registry metadata 替代本地 evidence
```

建议加入硬规则：

```text
online finding 必须标记 source = online
online evidence 必须显示 fetchedAt
online enrichment 不得降低 verdict
online enrichment 只能增加 evidence 或增加 risk
```

也就是：

```text
offline core remains source of truth
online data is advisory enrichment
```

否则产品会被质疑：

```text
你只是查 npm / GitHub metadata，然后假装懂安全。
```

## 风险 6：safe MCP Probe 不能轻易进入“真执行”

你列的 v0.3 P1 是 MCP Probe safe mode，仅 initialize + list_tools。

方向对，但危险点在于：

```text
启动 MCP server 本身就可能执行代码
```

所以不能把“只 list_tools”理解成安全。

真正的第一版 safe probe 应该分两层：

```text
Probe Interface v0: mock adapter + static metadata input
Probe Runtime v1: explicit --probe + sandbox requirement
```

也就是 v0.3 可以先做：

```text
packages/mcp-probe 接口
mock MCP server fixture
tool metadata schema
prompt-surface integration
```

但不要默认启动真实 server。

因为 Claude Code 和 MCP 的价值就是让 Claude 连接外部工具、数据库和 API，直接读取或操作系统；这也意味着只要启动了某些 server，就已经进入了更高风险边界。([Claude API Docs][3])

## 风险 7：Prompt Surface 应该升级为主线，而不是附属 detector

当前 prompt poisoning 已经有 detector，但 v0.3 应该把它上升为核心模块。

原因很简单：

```text
传统安全扫描器能扫 exec / fs / secrets
但 agent-native 工具必须扫 tool metadata 对模型行为的影响
```

2026 年关于 MCP threat modeling 的研究把 tool poisoning 描述为重要的 client-side 风险，并提出静态 metadata 分析、模型决策路径跟踪、行为异常检测和用户透明机制等防御方向。([arXiv][4])

所以 MCPGuard 的差异化不是：

```text
我也能扫 child_process.exec
```

而是：

```text
我能扫描进入模型上下文的 tool surface。
```

v0.3 应扩展扫描面：

```text
README.md
SKILL.md
MCP tool descriptions
resource descriptions
prompt templates
HTML comments
markdown comments
unicode invisible chars
schema field descriptions
```

## 风险 8：IDE 还没进场，开发者路径还不完整

CLI 是对的，但 IDE 是下一层价值。

VS Code 官方扩展 API 本身就是用来扩展编辑体验、构建诊断、hover、命令等能力的；Cursor 兼容 VS Code 插件生态，所以先做 VS Code diagnostics JSON 是合理路径。([Visual Studio Code][5])

但不要一上来做完整插件。

最优 v0.3 是：

```text
mcpguard diagnostics --json .cursor/mcp.json
```

输出 VS Code-style diagnostics：

```json
[
  {
    "file": ".cursor/mcp.json",
    "range": {
      "start": { "line": 7, "character": 10 },
      "end": { "line": 7, "character": 42 }
    },
    "severity": "error",
    "message": "BLOCK: Broad filesystem access",
    "code": "files.broad-path"
  }
]
```

这样之后插件只是消费 JSON，不重写扫描逻辑。

# 新增总原则：Auditability by Design

以后所有项目都必须默认满足：

```text
1. 结构清晰：目录、模块、边界一眼能看懂
2. 职责单一：每个 package / function / rule 只做一件事
3. 规则可解释：任何安全裁决都必须有证据链
4. 测试可复现：每个关键行为都有 fixture / golden / regression test
5. 权限最小化：默认不扩大权限，不执行未知代码，不读取真实 secrets
6. 变更可追踪：关键设计必须 ADR，关键规则必须 Rule Card
7. 输出可验证：JSON schema、SARIF、HTML、CLI 输出必须稳定
8. 错误可定位：失败时能明确知道是哪一层、哪条规则、哪个输入导致
9. 安全边界清楚：什么做、什么不做、什么需要用户授权，必须写明
10. 可回滚：每个 phase atomic commit，出错能退回
```

# MCPGuard 里应该立刻补进的原则

我建议在 `CLAUDE.md` 增加一节：

```md
## Auditability by Design

MCPGuard is a security-adjacent developer tool. Its own codebase must be easier to audit than ordinary application code.

Rules:

1. Every module must have a clear responsibility.
2. Every detector must produce evidence-backed findings.
3. Every security rule must have a rule card, tests, and positive/negative fixtures.
4. Every verdict-changing behavior must be covered by golden tests.
5. Every schema change must be explicit and backward-aware.
6. Every risky implementation decision must have an ADR.
7. No hidden magic, implicit side effects, or untraceable heuristics.
8. No LLM-only verdict path.
9. No network access in offline core.
10. No host execution of unknown MCP servers.
11. No weakening rules to pass tests.
12. No deleting fixtures to pass tests.
13. Prefer boring, readable code over clever abstractions.
14. Prefer explicit data flow over global state.
15. Prefer deterministic behavior over convenience.
```

# 我对这个补充的判断

这是必须加的。

因为 MCPGuard 未来如果要成为 trust layer，它自己不能像普通 MVP 那样粗糙。

```text
你是否误报？
你是否漏报？
你的规则是否黑盒？
你是否偷偷联网？
你是否执行了用户不知情的代码？

你的 SAFE / REVIEW / BLOCK 是否可解释？
```

所以代码必须从第一天就像安全基础设施，而不是普通创业项目。

最终原则可以压成一句话：

> **我们不是只扫描别人的风险；我们自己的代码也必须经得起扫描。**
