# 1. 我们最初的目的是什么？

我们一开始讨论的不是单纯做一个 CLI 工具，而是一个更大的方向：

> **Agent 时代的 trust layer。**

也就是：

```text
当 AI agent 要调用第三方 skill / MCP server / tool / API 时，

它是否可信？
它能读什么？

能花钱吗？

```

最早我们讨论过 Gateway / Payment / Marketplace，但后来推翻了直接从 gateway 开始的路径。

```text

需求侧真正先痛的是“不知道这些 tool 能不能信”
平台没有 trust data，gateway 没有说服力
```

所以我们选择了正确的 0→1 切入点：

```text
Scanner-first
→ Trust Index
→ Badge / Report
→ Team Policy
→ Runtime Guard
→ Registry
→ Gateway
→ Payment / Marketplace
```

当前完成的是第一块核心地基：

> **MCPGuard：AgentTrust 的本地风险裁决引擎。**

# 一、最优分发策略

我建议采用 **四层分发**。

```text
Layer 1 — GitHub Private Repo
用于源码托管、审计、CI、自测、Release artifact。

Layer 2 — GitHub Release Bundle
用于 v0.3 前的 bundle 下载与 smoke 验证。

Layer 3 — npm Package
用于真正的用户调用：npx mcpguard scan ...

Layer 4 — GitHub Actions / SARIF
用于团队 CI、安全扫描、PR 阻断、Code Scanning。
```

当前最优顺序是：

```text
先 GitHub private
→ 打通 GitHub CI
→ 打通 npm pack / local install
→ 再接 npm publish
→ 最后 public release
```

不要现在急着 public。

理由：

```text
1. 你还没有 npx mcpguard 打通
2. 真实 corpus 还没开始
3. SECURITY.md 还没补
4. 公开后别人会按安全工具标准审查你
5. 先 private + CI + release artifact 更稳
```
